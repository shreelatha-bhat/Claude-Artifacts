# Work Item Quality Audit

## Purpose

Evaluate the quality of a work item description based on Agile Operations standards.

The goal is to ensure clarity, completeness, and alignment with value delivery by validating:

- Job to Be Done (JTBD)
- Outcomes
- Dependencies
- Definition of Done (DoD)

---

## When to Use

Use this skill when:

- Reviewing newly created work items
- Auditing backlog quality
- Identifying vague or incomplete descriptions
- Validating if a work item is ready to be picked up
- Checking for AI-generated or unclear content

---

## Input

You will be given a work item description.

---

## Instructions

1. Carefully read the work item description.
2. Evaluate whether the following are clearly defined:

### 1. Job to Be Done (JTBD)
- Is the purpose of the work clearly stated?
- Is it clear what problem is being solved and why?

### 2. Outcomes
- Are expected results or impact clearly mentioned?
- Does it describe what success looks like?

### 3. Dependencies
- Are dependencies identified (people, systems, approvals)?
- Are any blockers or prerequisites mentioned?

### 4. Definition of Done (DoD)
- Are completion criteria clearly defined?
- Can someone verify when the work is complete?

---

## Evaluation Criteria

For each section (JTBD, Outcomes, Dependencies, DoD):

- **Clear** → Well-defined and actionable
- **Partial** → Some information present but incomplete
- **Missing** → Not mentioned or unclear

---

## Additional Checks

- Identify vague or generic language
- Identify overly broad or unclear scope
- Detect AI-generated or template-like content that lacks real clarity
- Check if the work item is actionable without further clarification

---

## Output Format

### Work Item Quality Summary

- Overall Quality: High / Medium / Low

---

### Section-wise Evaluation

**JTBD:**
> (Clear / Partial / Missing + explanation)

**Outcomes:**
> (Clear / Partial / Missing + explanation)

**Dependencies:**
> (Clear / Partial / Missing + explanation)

**Definition of Done:**
> (Clear / Partial / Missing + explanation)

---

### Key Issues

- List specific problems in the description

---

### Recommendations

- Suggest clear improvements:
  - What needs to be added
  - What needs to be clarified
  - What should be rewritten

---

### Final Assessment

- Is this work item ready to be picked up? (Yes / No)
- If not, what is missing?

---

## Writing Style

- Be direct and practical
- Avoid generic feedback
- Focus on clarity and usability
- Make feedback actionable
